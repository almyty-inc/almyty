# Versioning and Rollback

almyty tracks how an agent changes over time in two layers: an **audit log**
that records every change automatically, and **version snapshots** you save as
named restore points and can roll back to.

## Audit log

Every change to an agent — created, configuration updated, tools changed,
activated, a version saved, a rollback, an invocation — is recorded
automatically, with the timestamp, the user, and what changed. It is the live
history of the agent; nothing to turn on.

![The Audit Log on an agent — every change tracked with who, when, and what](/screenshots/versioning-audit.png)

| Event | Description |
|-------|-------------|
| `created` | Agent was created |
| `updated` | Configuration, pipeline, or tools were modified |
| `activated` / `deactivated` | Status changed |
| `version_saved` | A version snapshot was created |
| `rolled_back` | Pipeline was restored from a previous version |
| `invoked` | Agent was executed |

## Version snapshots and rollback

When you are about to make a big change, save a snapshot first. On the agent's
**Versions** tab, click **Save Version** and write a short changelog; almyty
stores the full configuration with an auto-incremented version number. To
restore one, find it in the list and click **Rollback** — the agent's
configuration is replaced with that snapshot, and the rollback itself is recorded
in the audit log. Rolling back never deletes newer versions, so you can roll
forward again at any time.

Save a version over the API:

```bash
curl -X POST https://api.almyty.com/agents/{id}/versions \
  -H "Authorization: Bearer $TOKEN" \
  -d '{ "changelog": "Added a Gemini reviewer to the verify panel" }'
```

## Good practice

- Save a version before restructuring an agent, so there is always a restore point.
- Write a real changelog — it is what makes the history useful later.
- Test the agent after a rollback to confirm the restored configuration behaves.
