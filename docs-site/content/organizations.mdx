# Organizations

Organizations are the top-level tenant boundary in almyty. Every resource -- APIs, tools, gateways, agents, and LLM providers -- belongs to exactly one organization. Organizations provide multi-tenancy, team collaboration, and role-based access control.

## In the UI

1. Click your organization name in the sidebar footer to open the org switcher
2. Select an organization or click **Create Organization**
3. To manage members, navigate to **Settings** and open the **Members** tab
4. Click **Invite Member**, enter their email and role, and click **Send**
5. To create a team, open the **Teams** tab, click **Create Team**, and add members

## Via the API

### Create an organization

```bash
curl -X POST /organizations \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Acme Corp",
    "description": "Main production organization"
  }'
```

### List members

```bash
curl /organizations/{orgId}/members \
  -H "Authorization: Bearer $TOKEN"
```

### Invite a member

```bash
curl -X POST /organizations/{orgId}/members \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "email": "developer@acme.com",
    "role": "member"
  }'
```

### Update a member's role

```bash
curl -X PATCH /organizations/{orgId}/members/{userId} \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "role": "admin"
  }'
```

### Remove a member

```bash
curl -X DELETE /organizations/{orgId}/members/{userId} \
  -H "Authorization: Bearer $TOKEN"
```

### Delete an organization

Only the owner can delete an organization. This permanently removes all resources and cannot be undone.

```bash
curl -X DELETE /organizations/{orgId} \
  -H "Authorization: Bearer $TOKEN"
```

## Roles

| Role | Description |
|------|-------------|
| `owner` | Full access. Can delete the organization and manage billing. |
| `admin` | Manage members, APIs, tools, gateways, and agents. |
| `member` | Create and manage own resources. Invoke agents. |
| `viewer` | Read-only access to all resources. |

### Permission matrix

| Action | Owner | Admin | Member | Viewer |
|--------|-------|-------|--------|--------|
| View resources | Yes | Yes | Yes | Yes |
| Create resources | Yes | Yes | Yes | No |
| Edit own resources | Yes | Yes | Yes | No |
| Edit all resources | Yes | Yes | No | No |
| Manage members | Yes | Yes | No | No |
| Delete organization | Yes | No | No | No |

## Teams

Teams are sub-groups within an organization for finer-grained access control.

### Create a team

```bash
curl -X POST /organizations/{orgId}/teams \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Backend Team",
    "description": "Backend API development"
  }'
```

### Add a team member

```bash
curl -X POST /organizations/{orgId}/teams/{teamId}/members \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "userId": "user-uuid",
    "role": "member"
  }'
```

### Team roles

| Role | Description |
|------|-------------|
| `lead` | Can manage team members and team settings |
| `member` | Standard team member |

## Switching organizations

Users can belong to multiple organizations. The frontend stores the current organization in app state. API requests are scoped to the organization associated with the JWT token.

To work with a specific organization's resources, use organization-scoped endpoints:

```bash
curl /organizations/{orgId}/tools \
  -H "Authorization: Bearer $TOKEN"
```
