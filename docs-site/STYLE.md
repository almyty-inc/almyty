# almyty docs & content writing style

The brief for anyone (human or agent) writing user-facing docs, marketing
copy, READMEs, or site content. Reference docs (api-reference) relax a few
rules where noted. Keep this short; if you're adding to it, cut something.

## Voice

Write for a user trying to *get something done*, not to document internals.
Confident, concrete, concise. No marketing fluff, no hedging, no oversell.
State what a thing does and show it working.

## The hard rules

- **`almyty` is always lowercase.** Capitalize `Almyty` only at the start of
  a sentence. Never mid-sentence, never in a heading that isn't sentence-start.
- **No em-dashes.** They read as AI slop. Use a colon, a comma, parentheses,
  or two sentences. (`foo: bar`, not `foo — bar`.)
- **No emoji.** Anywhere. Not in headings, not as bullets, not in callouts.
- **No AI attribution.** Never reference AI tools, "generated by", session
  IDs, or co-author trailers in content or commits.

## Structure: payoff first

- **Lead with the outcome, then the how.** Open a page with what the reader
  gets and a picture of it working, not with architecture or a concept map.
- **Follow the real user journey**, not the internal pipeline. The journey is:
  install the CLI, turn an API into tools, publish a gateway, see it running
  inside Claude Code (or any MCP client). Show the payoff up front.
- **Visual over prose.** Prefer a real screenshot or a short command +
  its real output to a wall of text. Every screenshot referenced must exist.
- **Cut ruthlessly.** A page that was 247 lines is usually better at 90.
  Delete restating, delete "in this section we will", delete throat-clearing.

## Commands & code

- **No `curl` blocks in user docs.** Show the CLI (`npx @almyty/...`) or the
  in-product action. Raw `curl`/HTTP belongs only in `api-reference`.
- Every command must actually run and produce the output shown. No invented
  flags, no stale package names, no pre-transfer URLs
  (`github.com/almyty-inc/almyty`, not `github.com/almyty/almyty`).
- Keep code/config in sync with reality: ports, image tags, versions must
  match the real `docker-compose.yml` / manifests, not aspirational values.

## Brand quick-ref

- Primary violet `#8b5cf6` (dark) / `#7C3AED` (light) — not indigo.
- Cyan accent `#22d3ee` / `#0891B2`. Protocol badges: MCP=violet, A2A=cyan,
  UTCP=emerald, SOAP=amber, GraphQL=rose, REST=blue.
- Fonts: Manrope (headings), DM Sans (body), JetBrains Mono (code).

## Before you open a docs PR

- Zero em-dashes (`grep -n "—"` your files).
- Balanced code fences; every image path resolves.
- `almyty` lowercase throughout.
- Claims match the shipped product (features, flags, versions, ports).
