# QA Tour Findings — 2026-06-11 staging

Browser walkthrough of `app.staging.almyty.com` against a freshly registered
account (`qa-tour-1781199750@almyty.test` / org `QA Tour Org`). This tour seeded
real data — a Petstore API import (19 generated tools), one HTTP tool, one
sandboxed-JavaScript tool, four gateways (MCP, UTCP, A2A, plus the system
`almyty` gateway), and a Pet Finder workflow agent — then drove live traffic
across all three protocols so the analytics dashboards would show populated
numbers instead of zeros.

The good news first: the three medium/low UI bugs from the 2026-06-03 tour are
fixed. Sidebar highlighting is correct (only the current page is active),
gateway/runner counts pluralize ("1 gateway", not "1 gateways"), and the
sign-up form now shows an inline "You must accept the terms and conditions"
error instead of failing silently. Five new issues found, ordered by severity.

## 1. Gateway detail: generated API keys never appear in the "API Keys" list

**Severity:** high — the panel claims a gateway has no keys when it has several, and a key generated from the UI vanishes the moment its one-time reveal dialog closes.

On a gateway with API-key auth configured, the **Authentication → API Keys**
panel always renders "No API keys yet. Generate one to allow clients to access
this gateway." even when keys exist. I confirmed two keys on the Petstore MCP
gateway via `GET /gateways/:id/auth/api-keys` (200, two entries in `data[]`),
and the browser's own network log shows that same 200 response with both keys
reaching the page. Generating a third key through the UI works — the success
dialog shows the `gw_…` secret once — but after clicking **Done** the list
still says "No API keys yet".

So the data is present in the response and the request is firing on mount; the
list just never renders it. The render guard is `hasApiKeyAuth && keys.length`,
and `keys` comes out empty despite the payload. Most likely the deployed bundle
is reading the wrong shape (the response is `{success, data:[…], message}` and
the keys live in `data`, not a `keys` field), or the list is gated on an auth
config the panel doesn't think is "api_key". Either way, anyone managing keys
through the UI is told they have none and can't see or revoke the ones they do.

Screenshot: `gateway-api-keys-not-listed.png` (panel reads "No API keys yet"
with API-key auth active).

## 2. Analytics Overview: "Requests" tiles stay at 0 while every other surface counts the same traffic

**Severity:** medium — the headline request counters read 0 even under heavy load, which makes the Overview tab look broken next to the populated tabs beside it.

After driving ~400 protocol requests (MCP, UTCP, A2A), the Overview tab shows
**Requests (24h) = 0** and **Requests (7d) = 0**, with the 7-day chart flat at
zero. Meanwhile the same page's other tiles are correct — Tool Executions reads
67, Audit Trail reads 77 actions — and the **Gateways** tab correctly attributes
the traffic per protocol (Petstore MCP 16, Petstore UTCP 3, Pet Finder A2A 1,
all 100% success). The gateway list page also shows "355 (354 ok)" requests for
the MCP gateway. So the traffic is being counted everywhere except the Overview
request tiles.

Root cause is a source mismatch. The Overview request counters come from
`request_logs` filtered on `metadata->>'protocol' IS NOT NULL`, but the
multi-tenant protocol traffic goes through the unified endpoint
(`/:orgSlug/:resourceSlug`), and the request-logging interceptor's
`detectProtocol()` only tags paths that literally contain `/mcp`, `/utcp`, or
`/a2a`. A request to `/qa-tour-org-.../petstore-mcp` matches none of those, so
its log row (if written at all) has a null protocol and is filtered out. The
per-gateway and per-tool stats come from the semantic `usage_metrics` counters
(`mcp_tool_call`, `utcp_manual`, `a2a_message`, …) which are emitted correctly,
which is why those surfaces are right. The Overview tiles should read from the
same semantic counters, or `detectProtocol()` needs to recognize unified-endpoint
paths.

Screenshots: `analytics-overview-requests-zero.png` (Requests 0, Tool
Executions 67) and `analytics-request-log-management-only.png` (Request Log
shows only management-API rows, all with protocol "--").

## 3. Analytics Request Log shows only management-API calls, never protocol traffic

**Severity:** medium — the tab documented as "every gateway request" is in practice a log of dashboard API calls.

Closely related to #2. The **Request Log** tab lists rows like
`GET /gateways/:id/auth/api-keys` and `GET /gateways/:id` — the frontend's own
calls to the management API — each with Protocol "--". None of the actual MCP /
UTCP / A2A protocol requests appear, even right after generating hundreds of
them. The docs describe this tab as "every gateway request with … protocol,
status code, latency, and caller IP", so as shipped it's misleading: it logs the
wrong class of request and never populates the Protocol column. Same fix as #2 —
protocol detection on unified-endpoint paths.

Screenshot: `analytics-request-log-management-only.png`.

## 4. Onboarding checklist shows stale "1 of 4 complete" after switching to an empty org

**Severity:** low — false-positive guidance, only on org switch.

Create a second, empty organization and switch to it via the sidebar org
switcher (without a full page reload) and the Dashboard's "Welcome to almyty"
checklist still reads **1 of 4 complete** with "Connect your first API" struck
through — carrying over the completion state from the previous org. A hard reload
fixes it (the empty org then correctly shows "0 of 4 complete", nothing struck
through), so the onboarding query isn't re-scoped to the new org on switch; it
keeps the cached result keyed to the old org until the page reloads.

This is the mirror image of last tour's finding #4 (which was about counting
system gateways). That one is fixed — a genuinely fresh org now shows 0/4 — but
the in-app org switch doesn't invalidate the onboarding state.

Screenshot: `org-switch-stale-onboarding.png` (empty "QA Fresh Org" showing
1 of 4 complete).

## 5. UTCP and A2A gateways always report "0 requests" in the gateway list

**Severity:** low — cosmetic count mismatch; the traffic is recorded elsewhere.

On `/gateways`, the REQUESTS column reads "0 requests" for the Petstore UTCP and
Pet Finder A2A gateways even after they served live traffic, while the Petstore
MCP gateway correctly shows "355 (354 ok)". The Analytics → Gateways tab counts
all three protocols fine for the same period (UTCP 3, A2A 1), so the data exists;
the gateway-list REQUESTS column is reading a counter that's only wired up for
MCP. Either it's pulling from `request_logs` (which only has MCP rows, per #2) or
the per-gateway request stat helper only increments on the MCP path.

Screenshot: `gateways-utcp-a2a-zero-requests.png`.

## 6. Detail-page header overflows on mobile

**Severity:** low — cosmetic; action buttons clip off the right edge at phone widths.

At a 390px-wide viewport, the gateway detail header lays the back arrow, the
gateway icon, the title, and the **Edit Gateway** button out on a single row, so
"Edit Gateway" runs off the right edge. The Authentication card's **Generate
Key** / **Add Auth Method** buttons clip the same way. The page is still usable
(the content below wraps fine) but the primary action on the page is partly
off-screen. The header should stack or wrap its actions below the title on narrow
viewports. Visible in `mobile/gateway-detail-tools.png` in the docs screenshot
set.

## What worked well

- Schema import auto-generated 19 typed tools from the Petstore OpenAPI spec.
- The sandboxed-JavaScript tool (`pet_roster`) executed correctly via the in-UI **Test Tool** runner (95ms) and over MCP `tools/call`, UTCP `execute`, and an agent `tool_call` node — same code, three protocols.
- The workflow agent (input → tool_call → output) ran to `completed` in ~140ms across 11 invocations, 100% success on the Analytics → Agents tab.
- Rate limiting works on the `@Throttle`-decorated sub-path routes: a burst of 80 requests to the UTCP manual endpoint returned 60×200 then 20×429. Unauthenticated requests with a bogus API key were correctly rejected 12/12 with 403.
- Per-protocol attribution on the Analytics → Gateways and → Tools tabs is accurate and now shows real numbers (the point of this tour).

## Out of scope this tour

- Agent builder internals (CLAUDE.md flags `agent-builder.tsx` / `agent-detail.tsx` as no-touch). The builder canvas was opened and screenshotted but not edited.
- Real OAuth / third-party credential flows (no live creds available — the LLM provider and credential created here are placeholders).
- The platform-admin `/monitoring/metrics` endpoint (returns 401 "disabled on this instance" on staging by design; the protocol/security counters it would expose are also surfaced through the analytics tabs).
- Global throttle (`RATE_LIMIT_MAX`): a 150-request burst to a non-decorated route never tripped, but staging may run a high limit — not pursued.
