### Execution Pipeline (Step by Step)

**Entry point:** `ToolExecutorService.executeTool(toolId, parameters, options)` at line 76 of `tool-executor.service.ts`.

**Full call chain:**

1. **Load tool** (line 88): Fetches tool with `['operation', 'operation.api', 'inputSchema', 'outputSchema']` relations.
2. **Status check** (line 97): Tool must be `ACTIVE`.
3. **Permission check** (line 102): If `userId` is provided, validates `use_tools` permission. For MCP unauthenticated sessions (`userId = null`), tool execution is allowed.
4. **Parameter validation** (line 116): Calls `validateParameters()` which uses `inputSchema.validate()` or falls back to checking `parameters.required` array.
5. **Parameter sanitization** (line 122): `sanitizeToolParameters()` checks for injection attacks, blocks dangerous inputs.
6. **Integrity verification** (line 132): If `definitionHash` exists, verifies tool definition hasn't been tampered with.
7. **Rate limit check** (line 141): Uses Redis for per-minute and per-hour counters per tool+user.
8. **Cache check** (line 157): If caching enabled, looks up Redis for cached result.
9. **Dispatch by type:**
   - **LLM tools** (line 178): If `tool.llmConfig` is present, interpolates `{{param}}` in promptTemplate, calls LLM provider.
   - **Custom code tools** (line 277): If `tool.code` is present, delegates to `CustomCodeExecutorService.executeCode()`.
   - **API-based tools** (line 326): Calls `executeOperation()` with retries and exponential backoff.

10. **`executeOperation()` dispatcher** (line 412): Routes by `api.type` (NOT by `tool.executionMethod`):
    - `ApiType.OPENAPI` -> `executeRestOperation()`
    - `ApiType.GRAPHQL` -> `executeGraphQLOperation()`
    - `ApiType.SOAP` -> `executeSOAPOperation()`
    - `ApiType.GRPC` -> `executeProtobufOperation()`
    - `ApiType.OTHER` -> **throws `Unsupported API type`**

**CRITICAL FINDING:** The dispatcher uses `api.type`, not `tool.executionMethod`. This means:
- Manual tools with `executionMethod = 'http'` but no operation/api will fail because `operation` is null.
- The `tool.executionMethod` field is set on manual tools but NEVER consulted by the executor for API-based routing.
- Manual HTTP/GraphQL/SOAP tools work ONLY because they generate JavaScript code at creation time (see frontend `createToolMutation` in `tools.tsx` lines 241-296), which is then executed as a custom code tool via `tool.code`.

**HTTP execution flow** (`executeRestOperation`, line 434):
1. SSRF validation on `api.baseUrl`
2. URL construction: `api.baseUrl + operation.endpoint`
3. Parameter grouping: Detects if params are pre-grouped (`{path, query, header, body}`) or flat
4. For flat params: extracts `{paramName}` from endpoint URL template for path params; rest go to query (GET) or body (POST/PUT/PATCH)
5. Path parameter substitution: `url.replace({key}, encodeURIComponent(value))` at line 497
6. Full URL SSRF re-validation
7. Header sanitization
8. Axios config built with: method, url, timeout, maxContentLength (10MB), params, paramsSerializer, headers (`Content-Type: application/json`)
9. Authentication applied via `addAuthentication()`
10. Body data added for POST/PUT/PATCH
11. Request sent via axios
12. Response returned as-is (`response.data`) -- NO response mapping, transformation, or field extraction

**Authentication application** (`addAuthentication`, line 824):
1. First tries to find a `Credential` entity linked to the API (`apiId + organizationId + isActive`)
2. If found, calls `applyCredential()` which:
   - For OAuth2: attempts token refresh if expired via `CredentialService.refreshOAuthToken()`
   - Calls `credential.getAuthHeaders()` (handles api_key, bearer, basic, jwt, custom)
   - Calls `credential.getQueryParams()` (handles api_key in query)
3. If no Credential found, falls back to `api.authentication` field (legacy inline config)

**Response handling:** Minimal. Returns `response.data` directly. No response mapping, no field extraction, no pagination handling, no success/failure condition checking beyond HTTP status.

### Feature 2: Enhanced HTTP Execution Pipeline

**What exists:**
- Basic HTTP execution in `executeRestOperation()` (lines 434-591)
- URL construction with path parameter substitution (`{param}` replacement)
- Query parameter serialization (including array support)
- JSON body sending
- Auth application (Credential entity or legacy api.authentication)
- SSRF protection
- Response is returned as raw `response.data`

**What's partially implemented:**
- `GatewayTool.transformations` has `inputMapping` and `outputMapping` but these are simple key remapping, not JSONPath/dot-path extraction
- Header injection from `operation.parameters.header` is extracted but sanitized

**What's completely missing:**
1. **Response mapping/extraction:** No way to extract a specific field from the response (e.g., `response.data.results` for pagination). `response.data` is returned wholesale.
2. **Pagination handling:** No cursor/offset/link-header pagination support. Each tool call returns one page.
3. **Body encoding beyond JSON:** Only `Content-Type: application/json` is set (line 537). No `application/x-www-form-urlencoded`, no `multipart/form-data`.
4. **Success conditions:** No way to define what constitutes success beyond HTTP 2xx. No custom success/failure detection.
5. **Response transforms:** No JSONPath, jq-style, or template-based response transformation.
6. **Request body templates:** Body is passed as-is from parameters or as the entire parameters object. No body template interpolation.
7. **Dynamic URL construction:** While path params are substituted, there's no way to construct URLs from parameter combinations beyond what `{param}` provides.
8. **Header parameter injection:** Headers from operation parameters are passed through, but manual tools can't define custom headers per-request.
9. **Retry configuration per-error-type:** Retries use exponential backoff but don't distinguish between retryable errors (5xx, timeout) and non-retryable (4xx).

**Where each enhancement slots in:**
- Response mapping: After `await axios(config)` in `executeRestOperation()`, apply a configurable extraction path
- Pagination: New method alongside `executeRestOperation()` that loops, detects next page, and aggregates
- Body encoding: Modify Content-Type and body serialization in `executeRestOperation()` based on tool/operation config
- Success conditions: After response, evaluate configurable conditions before marking as success/failure
- All configs would live in the `Tool.configuration` or a new `Tool.httpConfig` JSON field

## Document 4: Open Questions

1. **Should `ApiType.OTHER` be repurposed or should a new `ApiType.HTTP` be added?**
   - Repurposing: Fewer changes, `OTHER` is already the default. Risk: existing `OTHER` APIs (if any) might break.
   - New type: Cleaner semantics, no backward compatibility risk. Needs enum migration.
   - **Recommendation:** Add `ApiType.HTTP`. It's more explicit and `OTHER` can remain as a catch-all.

2. **Should manual HTTP tools still require an Api entity?**
   - Yes (recommended): Shared baseUrl and auth config across tools. Consistent model. The Api acts as a "connection" or "integration" container.
   - No: Simpler for one-off tools, but loses auth/config sharing.
   - **Compromise:** Make `apiId` on the tool optional for HTTP tools. If linked to an Api, inherit baseUrl/auth. If standalone, the tool's `httpConfig` must have the full URL.

3. **How should `httpConfig.path` combine with `api.baseUrl`?**
   - If tool has `httpConfig.path` = `/users/{id}` and linked Api has `baseUrl` = `https://api.example.com/v1`, final URL = `https://api.example.com/v1/users/{id}`.
   - If tool has no linked Api, `httpConfig.path` must be a full URL.
   - This mirrors how `operation.endpoint` works today.

4. **Should the Tool entity's `code` field be cleared when `httpConfig` is set?**
   - Probably yes, to avoid confusion. A tool should execute via ONE path: code, llmConfig, httpConfig, or operation-based.
   - The executor dispatch order should be: llmConfig -> code -> httpConfig -> operation.

5. **How deep should response mapping go?**
   - Simple dot-path (`data.results`) covers 80% of cases.
   - Full JSONPath (`$.data.items[*].name`) is more powerful but complex.
   - **Recommendation:** Start with dot-notation (e.g., `lodash.get(response.data, path)`), add JSONPath later.

6. **Should pagination be transparent to the caller or explicit?**
   - Transparent: Tool automatically fetches all pages, returns aggregated results. Simpler for AI agents but can be slow/expensive.
   - Explicit: Tool returns one page + cursor; agent decides whether to fetch more. More control, matches how APIs actually work.
   - **Recommendation:** Default to single-page with pagination metadata in the response. Add an `autoPageinate: boolean` flag for transparent mode.

7. **Tool Hub: global templates vs per-org templates?**
   - Global templates (shipped with the platform) are read-only, curated.
   - Per-org templates allow organizations to create their own reusable patterns.
   - **Recommendation:** Both. `organizationId = null` means global; non-null means org-specific.

8. **Should installing a template create an Api entity too, or just a Tool?**
   - If the template targets a specific API (e.g., "Slack - Send Message"), it should create both an Api (for baseUrl/auth) and a Tool.
   - This way, multiple tools from the same provider share one Api entity.
   - **Recommendation:** Templates can optionally reference a "provider config" that creates an Api. The install flow asks the user to either select an existing Api or create a new one.

### Stripe API
- **Custom HTTP API:** Create Api with baseUrl `https://api.stripe.com/v1`, type `HTTP`
- **Auth:** Bearer token (API key as bearer)
- **Tools:** e.g., "Create Customer" with `httpConfig: { method: 'POST', path: '/customers', bodyEncoding: 'form-urlencoded' }`
- **Pagination:** Stripe uses cursor-based with `starting_after` -- `pagination: { type: 'cursor', cursorParam: 'starting_after', cursorPath: 'data[-1].id' }`
- **Covered by:** `form-urlencoded` body encoding (critical for Stripe), cursor pagination, bearer auth
- **Note:** Without `form-urlencoded` support, Stripe API calls would fail. This is the most important body encoding addition.

### AWS S3 API
- **Custom HTTP API:** Create Api with baseUrl `https://s3.amazonaws.com`, type `HTTP`
- **Auth:** AWS Signature V4 -- this is NOT currently supported by ANY auth type
- **Tools:** e.g., "List Buckets" with `httpConfig: { method: 'GET', path: '/' }`
- **Gap:** AWS Signature V4 requires computing HMAC-SHA256 over canonical request, not just setting a header. This needs:
  - A new `CredentialType.AWS_SIGV4` or `CredentialType.CUSTOM` with a signature plugin
  - OR: Use the AWS SDK inside a custom code tool (which already works today)
- **Partially covered:** Would require either a new auth type or falling back to custom code tools. The `CredentialType.CUSTOM` with `headerName`/`headerValue` won't work because AWS auth requires computing the signature per-request.
- **Recommendation:** For V1, document AWS S3 as requiring custom code tools. Add AWS SigV4 auth in a later iteration.
